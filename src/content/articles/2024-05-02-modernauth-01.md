## Introdução
A internet é um espaço de interação e troca de informações entre diversos agentes, como usuários, aplicativos, serviços e recursos. Para garantir a segurança e a privacidade dessas interações, é necessário estabelecer mecanismos de autenticação e autorização que verifiquem a identidade e o acesso dos agentes envolvidos. Os protocolos modernos mais utilizados para esse fim são o **OpenID Connect(OIDC)**, **Open Authentication(OAuth)** e **Security Assertion Markup Language(SAML)**. Esses protocolos objetivam permitir que um aplicativo obtenha acesso limitado a um recurso protegido em nome de um usuário, sem que este tenha que compartilhar suas credenciais com o aplicativo.

O objetivo desta série de artigos é apresentar os conceitos e as aplicações dos protocolos **SAML**, **OAuth** e **OIDC** usando como IAM o [**Entra ID**](https://learn.microsoft.com/en-us/entra/identity/), explicando como eles operam, quais são os seus benefícios e desafios, e como eles podem ser implementados em diferentes cenários. A tese defendida neste artigo é que o **SAML**, **OAuth** e o **OIDC** são protocolos flexíveis, seguros e eficientes para a gestão de autenticação e autorização na internet, mas que também requerem cuidados e boas práticas para evitar vulnerabilidades e riscos.

O artigo foi divido em 6 seções:
* **Seção 1, será feita uma revisão teórica sobre os conceitos de autenticação e autorização e os protocolos mais utilizados atualmente separados em depois blocos legado e moderno.**
* Seção 2, será descrito o funcionamento do **SAML**, explicando os seus principais componentes, fluxos e especificações.
* Seção 3, será descrito o funcionamento do **OAuth**, explicando os seus principais componentes, fluxos e especificações.
* Seção 4, será descrito o funcionamento do **OpenID Connect**, explicando os seus principais componentes, fluxos e especificações.
* Seção 5, serão apresentados alguns exemplos de aplicações usando **SAML**, **OAuth** e **OIDC** em diferentes contextos, como redes sociais, serviços de nuvem e dispositivos móveis.
* Seção 6, serão discutidos os benefícios e os desafios do **SAML**,**OAuth** e **OIDC**, analisando as suas vantagens e desvantagens, bem como as suas recomendações e limitações.

### Conceitos de Autenticação e Autorização
A Autenticação(AuthN) é o processo de identificar o agente, ou seja responder a perguntar "Who are you?".

Já o processo de Autorização(AuthZ) é a validação se o agente tem da permissão de acesso aos recursos solicitados, "Can you do that?".

## Autenticação Legada
As autenticações legadas iremos definir como métodos tradicionais de autenticação do usuário sendo que normalmente esses métodos não possuem fatores adicionais de autenticação e é bem comum que as aplicações realizem autenticações ativas e não passivas.
Além desses métodos não terem sido desenhados para permitir autenticações de usuários localizados em outros IdP.

Temos alguns protocolos de autenticação legado bem conhecidos como: 
 
    * Basic Authentication 
    * Digest Authentication
    * NTLM
    * Kerberos
    
Nesse tipo de fluxo de autenticação normalmente as aplicações realizam todo o processo de autenticação o que aumenta a complexidade no desenvolvimento, manutenção, segurança e implementações de novas camadas de segurança.

### Basic Authentication
O Basic Authentication é o sistema de autenticação mais "básico / comum" do protocolo HTTP definido pela [RFC 7617](https://datatracker.ietf.org/doc/html/rfc7617). 

How it works:

![Basic Auth Flow, onde o cliente requisita acesso a pagina home e o servidor retornar o HTTP Code 401 de não autorizado solicitando a autenticação para o realm localhost. O cliente com essa informação captura as informações para autenticação e envia novamente a requisição com o header Authorization preenchido com Basic seguido do usuário e senha em Base64](/images/articles/modern-auth-01/basicAuth-Flow.png)

1. Quando um usuário tenta acessar um recurso protegido ele recebe um retorno do servidor com o código HTTP 401 (Unauthorized) e um cabeçalho WWW-Authenticate, que fornece informações sobre como se autenticar.

1. O cliente envia o pedido com o cabeçalho de autenticação no formato: 
     
    * Authorization: Basic {credenciais em base 64 no formato usuário:senha}. 
    
    >Lembre-se de que o Base 64 é um esquema de codificação, **não criptografia.**

1. Essas informações são recebidas no servidor, que verifica se as credenciais são válidas(podendo essa validação ser utilizado uma autenticação via LDAP ou local).

    * Se as credenciais estiverem correta o cliente receberá uma resposta diferente de 403 (Forbidden).

É importante observar que o uso do Base 64 se deve ao padrão MIME. Portanto, o Basic Authentication deve ser utilizado **apenas com uma conexão HTTPS([RFC2818 - HTTP over TLS](https://www.rfc-editor.org/rfc/rfc2818)) para garantir a segurança**. No entanto, mesmo com HTTPS, o Basic Auth não é 100% seguro, pois as credenciais são enviadas a cada requisição de autenticação nova.

Em resumo, o Basic Authentication é método simples e de rápida implementação, mas sua segurança depende do nível de risco dos dados em trânsito. 

Para dados sigilosos, outras opções de autenticação e autorização, como SAML, OAuth e OIDC, podem ser mais adequadas.

### Digest Authentication
A Digest Authentication, conforme definida na [RFC 2617](https://datatracker.ietf.org/doc/html/rfc2617), é um método pelo qual um servidor da web pode negociar credenciais, como nome de usuário ou senha, com o navegador da web de um usuário bem similar ao BASIC Authentication conform vimos anteriormente porém **diferentemente do Basic Authentication esse método calcula o HASH do usuário e senha antes de enviar ao site.** 

![Digest Auth Flow, o navegador solicita o acesso a uma página, o servidor retorna solicitando a autenticação do usuário. O navegador captura as informações de autenticação do usuário e devolve ao webserver qual irá validar essa autenticação em um DC autorizando o acesso](/images/articles/modern-auth-01/digestAuth-Flow.png)

**Processo de Autenticação:**
1. O cliente faz uma solicitação de acesso.

1. O servidor gera algumas informação dentre elas o nonce (código que será usado apenas uma vez) e o envia ao cliente.

1. O cliente calcula o hash da senha e outras informações (como o método HTTP e o URI solicitado) usando o nonce.

1. O cliente envia a resposta ao servidor no formato: Authorization: Digest username="usuário", realm="realm", nonce="nonce", uri="URI", response="hash".

1. O servidor verifica a resposta e concede acesso se tudo estiver correto.

Em resumo, a autenticação Digest é uma alternativa à autenticação básica, oferecendo maior segurança ao usar hashes e nonces para proteger as credenciais do usuário durante a comunicação com o servidor.

### NTLM

O NTLM (NT LAN Manager) é um protocolo de autenticação baseado em desafio-resposta. 

![NTLM Flow onde o usuário solicita acesso a um recurso porém o servidor do recurso gera um challenge e devolve para o cliente qual irá usar esse challenge para calcular o hash e enviar para o servidor do recurso poder validar o usuário](/images/articles/modern-auth-01/NTLM-Flow.png)

1. Protocolo NTLM:
    * O NTLM inclui várias versões, como LAN Manager versão 1 e 2, e NTLM versão 1 e 2.
    * Ele autentica usuários e computadores com base em um mecanismo de desafio/resposta.
    * O servidor ou controlador de domínio verifica se o usuário conhece a senha associada à conta.

1. Processo de Autenticação:
    * Quando um usuário tenta acessar um recurso protegido, o servidor gera um desafio.
    * O cliente (usuário ou computador) responde ao desafio com um hash da senha.
    * O servidor verifica o hash e concede acesso se for válido.

1. Aplicações Atuais:
    * O NTLM ainda é usado para autenticação em sistemas configurados como membros de um workgroup.
    * Também é usado para autenticação de logon local em servidores que não são controladores de domínio.
    * Apesar da recomendação no uso do Keberos como o método preferencial, aplicações podem fazer o uso do NTLM para autenticar seus usuários
    
Em resumo, o NTLM é um protocolo de autenticação legado que deve ser usado com cautela, considerando suas limitações e riscos de segurança.

### Kerberos

O Kerberos é um protocolo de autenticação de rede desenvolvido pelo Massachusetts Institute of Technology (MIT) na década de 1980. Ele fornece um método seguro para autenticar usuários e serviços em uma rede, garantindo que apenas usuários autorizados tenham acesso aos recursos.

![Kerberos Flow, o cliente ao tentar acessar um recurso ele irá obter os tickets necessários no TGS qual irá apresentar ao servidor da aplicação ](https://www.kerberos.org/images/krbmsg.gif)

A autenticação Kerberos é um processo de várias etapas que envolve os seguintes componentes:

* Cliente: O cliente inicia a necessidade de uma solicitação de serviço em nome do usuário.
* Servidor: O servidor hospeda o serviço ao qual o usuário precisa acessar.
* AS (Authentication Server): O AS executa a autenticação do cliente.
* KDC (Key Distribution Center) e seus três componentes:
* TGS (Ticket Granting Server): Emissor de bilhetes de serviço.
* Banco de Dados Kerberos: Armazena informações de autenticação.

Em resumo, o Kerberos oferece uma solução confiável para autenticação em redes distribuídas, garantindo a proteção dos dados confidenciais e a autenticação segura de usuários e serviços

## Introduction
The Internet is a space for interaction and information exchange between many actors — users, applications, services, and resources. To preserve the security and privacy of these interactions, we need authentication and authorization mechanisms that verify the identity and access rights of every actor involved. The most widely used modern protocols for this purpose are **OpenID Connect (OIDC)**, **OAuth (Open Authentication)**, and **SAML (Security Assertion Markup Language)**. They allow an application to obtain limited access to a protected resource on behalf of a user, without that user having to share credentials with the application.

The goal of this series is to introduce the concepts and applications of **SAML**, **OAuth**, and **OIDC** using [**Entra ID**](https://learn.microsoft.com/en-us/entra/identity/) as the IAM platform — how they work, what their benefits and challenges are, and how they can be implemented in different scenarios. The thesis defended here is that **SAML**, **OAuth**, and **OIDC** are flexible, secure, and efficient protocols for managing authentication and authorization on the Internet, but they require care and best practices to avoid vulnerabilities and risks.

* **Section 1 — theoretical review of authentication and authorization concepts and the most-used protocols today, separated into legacy and modern blocks.**
* Section 2 — how **SAML** works: components, flows, and specifications.

* Section 4 — how **OpenID Connect** works: components, flows, and specifications.

### Authentication and Authorization concepts
**Authentication (AuthN)** is the process of identifying the actor — answering the question *"Who are you?"*.

**Authorization (AuthZ)** is the validation of whether the actor has permission to access the requested resources — *"Can you do that?"*.

## Legacy authentication
We define legacy authentication as traditional user-authentication methods that typically lack additional factors, and where it's common for applications to perform active (not passive) authentication. These methods were not designed to allow authentication of users living in another IdP.

Some well-known legacy authentication protocols include:

    * Basic Authentication
    * Digest Authentication
    * NTLM
    * Kerberos

In this kind of flow, the application usually handles the entire authentication process itself, which increases the complexity of development, maintenance, security, and the rollout of new security layers.

### Basic Authentication
Basic Authentication is the most "basic / common" authentication system in HTTP, defined by [RFC 7617](https://datatracker.ietf.org/doc/html/rfc7617).

How it works:

![Basic Auth flow: the client requests access to the home page and the server returns HTTP 401 Unauthorized asking for authentication against the localhost realm. The client gathers the credentials and re-sends the request with an Authorization header containing 'Basic' followed by the username:password pair encoded in Base64.](/images/articles/modern-auth-01/basicAuth-Flow.png)

1. When a user tries to access a protected resource, the server returns HTTP 401 (Unauthorized) together with a `WWW-Authenticate` header that tells the client how to authenticate.

1. The client sends the request with an authentication header in the format:

    * `Authorization: Basic {credentials in Base64, in the form user:password}`.

    > Remember: Base64 is an **encoding** scheme, **not encryption**.

1. The server receives this information and validates the credentials (validation can use a local store or an LDAP backend).

    * If the credentials are correct, the client receives a response other than 403 (Forbidden).

It's important to note that Base64 is used because of the MIME standard. Therefore, Basic Authentication should be used **only over HTTPS ([RFC 2818 — HTTP over TLS](https://www.rfc-editor.org/rfc/rfc2818)) to guarantee transport security**. Even with HTTPS, Basic Auth is not 100% safe — credentials are sent on every new authentication request.

In short, Basic Authentication is simple and quick to implement, but its security depends on the sensitivity of the data in transit.

For sensitive data, other authentication and authorization options — such as SAML, OAuth, and OIDC — are typically more appropriate.

### Digest Authentication
Digest Authentication, as defined by [RFC 2617](https://datatracker.ietf.org/doc/html/rfc2617), is a method by which a web server can negotiate credentials — such as a username and password — with a user's browser. It's similar to Basic Authentication, but **unlike Basic, this method hashes the username and password before sending them to the server.**

![Digest Auth flow: the browser requests a page, the server responds asking for the user's credentials, the browser collects them and sends them back to the web server, which validates them against a DC and authorizes access.](/images/articles/modern-auth-01/digestAuth-Flow.png)

**Authentication process:**
1. The client makes an access request.

1. The server generates several pieces of information — including a nonce (a code to be used only once) — and sends them to the client.

1. The client hashes the password along with other information (such as the HTTP method and the requested URI) using the nonce.

1. The client sends the response in the format: `Authorization: Digest username="user", realm="realm", nonce="nonce", uri="URI", response="hash"`.

1. The server verifies the response and grants access if everything is correct.

In short, Digest is an alternative to Basic Authentication that offers more security by using hashes and nonces to protect user credentials during communication with the server.

### NTLM

NTLM (NT LAN Manager) is a challenge-response-based authentication protocol.

![NTLM flow: the user requests access to a resource, the resource server generates a challenge and returns it to the client, which uses that challenge to compute a hash and send it back so the server can validate the user.](/images/articles/modern-auth-01/NTLM-Flow.png)

1. The NTLM protocol:
    * NTLM includes several versions, such as LAN Manager v1 and v2, and NTLM v1 and v2.
    * It authenticates users and computers based on a challenge/response mechanism.
    * The server or domain controller verifies that the user knows the password associated with the account.

1. Authentication process:
    * When a user tries to access a protected resource, the server generates a challenge.
    * The client (user or computer) answers the challenge with a hash of the password.
    * The server verifies the hash and grants access if it's valid.

1. Current applications:
    * NTLM is still used for authentication on systems configured as members of a workgroup.
    * It's also used for local logon authentication on non-domain-controller servers.
    * Even though Kerberos is the recommended preferred method, applications can still use NTLM to authenticate their users.

In short, NTLM is a legacy authentication protocol that should be used with caution, given its limitations and security risks.

### Kerberos

Kerberos is a network authentication protocol developed by the Massachusetts Institute of Technology (MIT) in the 1980s. It provides a secure method for authenticating users and services on a network, ensuring that only authorized users can access resources.

![Kerberos flow: when the client tries to access a resource, it obtains the required tickets from the TGS and presents them to the application server.](https://www.kerberos.org/images/krbmsg.gif)

Kerberos authentication is a multi-step process involving the following components:

* Client: initiates a service request on behalf of the user.
* Server: hosts the service the user needs to access.
* AS (Authentication Server): performs the client authentication.
* KDC (Key Distribution Center) and its three components:
* TGS (Ticket Granting Server): issuer of service tickets.
* Kerberos Database: stores authentication information.

In short, Kerberos provides a reliable solution for authentication in distributed networks, ensuring the protection of confidential data and the secure authentication of users and services.
